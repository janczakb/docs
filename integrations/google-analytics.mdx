Filament Short URL sends `short_url_visit` events to Google Analytics 4 using the **GA4 Measurement Protocol API** â€” a direct server-to-server HTTP call made from your Laravel application. Because no JavaScript runs in the visitor's browser, ad blockers, cookie consent banners, and browser privacy settings have zero effect on your analytics data.

|  | Server-Side (this feature) | Client-Side (gtag.js) |

## Configuration

<Steps>
  <Step title="Create a Measurement Protocol API Secret">
    In Google Analytics, go to **Admin â†’ Data Streams â†’ (your stream) â†’ Measurement Protocol API secrets** and create a new secret. Copy the generated value.

    <Tip>
      You'll find Measurement Protocol API secrets under **Admin â†’ Data Streams â†’ choose your web stream â†’ Measurement Protocol API secrets**. Each secret is a short alphanumeric string â€” not the same as your Measurement ID (`G-XXXXXXXXXX`).
    </Tip>
  </Step>
  <Step title="Add your API Secret to the environment">
    Set the following variable in your `.env` file:

    ```env
    GA4_API_SECRET=your_measurement_protocol_secret_here
    ```

    Alternatively, enter it directly in **Settings â†’ Tracking â†’ Google Analytics 4 â†’ GA4 API Secret**.
  </Step>
  <Step title="Optionally set a Firebase App ID">
    If your GA4 property is linked to a Firebase project, set:

    ```env
    FIREBASE_APP_ID=1:1234567890:web:abc123def456
    ```

    If omitted, the plugin uses the Measurement ID from the per-link `ga_tracking_id` field as the stream identifier.
  </Step>
  <Step title="Set a GA Tracking ID on each link">
    Open the link's edit form and navigate to the **Tracking** tab. Enter your GA4 Measurement ID in the **GA Tracking ID** field (format: `G-XXXXXXXXXX`).

    Only links with a `ga_tracking_id` value configured will send events. Links without one are silently skipped.
  </Step>
</Steps>

## How Events Are Sent

As of **v5.2.0**, the GA4 Measurement Protocol pipeline lives in the dedicated **`Ga4MeasurementProtocolService`** singleton (extracted from `TrackShortUrlVisitJob`). When a visitor clicks a short URL that has a `ga_tracking_id` set, an asynchronous queued job calls the service which sends a payload to the GA4 collector.

### Full Payload Shape

| Field | Source |
| --- | --- |
| `client_id` | `sha256($ipAddress . $userAgent)` â€” privacy-safe, deterministic across visits |
| `timestamp_micros` | Visit timestamp in microseconds |
| `session_id` | Deterministic per-visitor session identifier |
| `engagement_time_msec` | Synthetic engagement window for GA4 session attribution |
| `events[]` | Three events per visit: standard `page_view`, standard `click`, custom `short_url_visit` |
| `event_params` | URL key, destination URL, country code, device type, referrer host, UTM fields |

Standard event names (`page_view`, `click`) are used so the visits appear in GA4's built-in reports â€” not just the custom event explorer.

### Measurement ID Validation

`Ga4MeasurementProtocolService::isValidMeasurementId()` enforces the regex `G-[A-Z0-9]+` before any HTTP request is opened. Invalid `ga_tracking_id` values are logged and skipped â€” they never consume a queue retry or count against your API secret rate limit.

### Test Connection Button

In **Settings â†’ Tracking â†’ Google Analytics 4** the **Test connection** button (added v5.2.0) validates your real Measurement ID \+ API secret against the GA4 debug collector via `validateCredentials()`. The placeholder `G-XXXXXXXXXX` is explicitly rejected so you can't accidentally save a stub.

<Info>
  Client IDs are computed as `hash('sha256', $ipAddress . $userAgent)`. No raw IP addresses or User Agents are ever sent to Google Analytics. This approach satisfies GDPR requirements for analytics without losing session coherence.
</Info>

## Troubleshooting

<AccordionGroup>
  <Accordion title="Events are not appearing in GA4 DebugView">
    Make sure your `GA4_API_SECRET` is the **Measurement Protocol API secret**, not your Measurement ID or Data API credentials. Also confirm the link's `ga_tracking_id` field is set and matches your GA4 property (`G-XXXXXXXXXX` format). Allow up to 60 seconds for DebugView to reflect events.
  </Accordion>

  <Accordion title="Events show up in DebugView but not in reports">
    GA4 Measurement Protocol events can take 24â€“48 hours to appear in standard reports. DebugView is the correct tool for immediate verification. Also check that your GA4 property is not in a data-processing hold.
  </Accordion>

  <Accordion title="I have a queue worker â€” do jobs run there?">
    Yes. GA4 hit jobs are dispatched to whichever queue connection you configure in **Settings â†’ General â†’ Queue Connection**. Ensure your worker is running with `php artisan queue:work`. If you use the `sync` driver, the GA4 call happens synchronously during the visit recording job, still without blocking the redirect.
  </Accordion>
</AccordionGroup>
